* 技术决策

** D001：主用 Emacs，但不使用配置发行版

状态：已确定。

不使用 Doom Emacs、Spacemacs 等发行版。原因不是它们不好，而是当前目标是理解 Emacs 的启动、buffer、keymap、package、Elisp 和 GPT 接入方式。发行版会隐藏大量默认行为，不利于学习和排错。

** D002：使用 =~/.config/emacs/= 作为配置根目录

状态：已实现。

当前 ~user-emacs-directory~ 是 =~/.config/emacs/=。这符合 XDG 风格，也和本机其他配置目录一致。

** D003：保留 ~early-init.el~ 与最小 ~init.el~ 的启动阶段边界

状态：已实现。

~early-init.el~ 负责启动早期行为，包括临时 GC、禁用 package 自动启动和关闭基础 UI 元素。 ~init.el~ 是正常初始化阶段的稳定引导文件，只从仓库本地离线加载 ~config.el~ ；正常配置的人工维护入口由 D022 规定。

这样做的原因：

- 减少 GUI 启动闪烁。
- 避免 package 系统在配置前自动加载。
- 保持启动阶段和正常配置阶段边界清晰。
- 即使 Literate 配置继续增长，也不把需要早于 package 初始化生效的职责混入 ~config.org~ 。

** D004：优先使用 Emacs 内置能力

状态：已确定。

当前优先使用内置 ~use-package~、内置编辑模式和内置历史能力。后续代码能力优先使用内置 ~eglot~ 和 ~treesit~ ，只有内置能力明显不足时再安装第三方包。

** D005：macOS 修饰键策略

状态：已实现。

当前配置：

- ~Option~ 作为 ~Meta~ 。
- ~Command~ 作为 ~Super~ 。

原因：Emacs 的 ~M-x~、~M-f~、~M-b~ 等操作需要稳定 Meta 键；macOS 的 Command 仍可作为独立 Super 修饰键，后续可绑定图形环境专用快捷键。

** D006：字体策略

状态：已实现。

当前策略：

- 英文和代码使用 17pt ~Fira Code~ 。
- 中文回退使用 ~PingFang SC~ 。

原因：代码字体和中文字体混排时，如果不显式设置中文回退，容易出现缺字、字宽不一致或显示风格混乱。

** D007：运行状态文件需要单独处理

状态：已实现。

运行状态与可清理缓存使用不同的 XDG 边界：

- history、recentf、places、project.el 项目、书签和 Customize 写入 =${XDG_STATE_HOME:-~/.local/state}/emacs/=。
- backup、autosave 和 auto-save-list 写入 =${XDG_CACHE_HOME:-~/.cache}/emacs/=。
- 第三方包 ~elpa/~ 继续保留在配置目录，但由 ~.gitignore~ 排除，并通过包清单重建。

原因：配置源码必须可安全提交和同步；持久状态不能因为清理缓存而丢失，可重建缓存也不应进入版本控制。

** D008：GPT 接入优先评估 ~gptel~

状态：计划。

后续 GPT 接入优先评估 ~gptel~ 。原因是它符合 Emacs 的 buffer 工作流，可以在任意 buffer 和选区中使用 LLM，不局限于独立聊天窗口。

约束：

- API key 不写进仓库。
- GPT 输出默认视为建议，不直接当作已验证事实。
- GPT 工作流必须服务于代码、笔记和复盘，不应替代基础编辑能力。

** D009：使用 Batppuccin Mocha 作为当前主题

状态：已实现。

当前主题从 Emacs 内置 ~modus-vivendi-tinted~ 切换为第三方 ~batppuccin~ 包中的 ~batppuccin-mocha~ 。

选择原因：

- ~batppuccin-mocha~ 是深色主题，视觉比内置 Modus 更柔和，更符合当前“先把界面弄好看”的目标。
- Batppuccin 项目把 Catppuccin 的四种 flavor 做成标准 Emacs theme，例如 ~batppuccin-mocha~、~batppuccin-macchiato~、~batppuccin-frappe~ 和 ~batppuccin-latte~ ，可以直接用 ~load-theme~ 加载。
- 包已发布到 MELPA，可以用 Emacs 内置 ~package.el~ 安装，不需要额外插件管理器。

约束：

- 这是当前第一个第三方 Emacs Lisp 包。
- ~elpa/~ 是可再安装依赖目录，已由 ~.gitignore~ 排除。
- 如果后续主题不适合中文、Org 或代码高亮，可以换回内置 Modus 或评估 ~ef-themes~、~doom-themes~ 。

** D010：启动过程禁止隐式联网安装

状态：已实现。

启动只加载本机已有包，不执行 ~package-refresh-contents~ 或 ~package-install~ 。当 Batppuccin 缺失或加载失败时，回退到 Emacs 内置 ~modus-vivendi-tinted~ ，并提示用户稍后显式运行 ~M-x package-install-selected-packages~ 。

原因：启动必须在离线、弱网和代理异常时仍然可用；安装依赖是明确的维护动作，不应成为隐藏的启动副作用。

** D011：GC 优化必须有启动阶段兜底恢复

状态：已实现。

~early-init.el~ 临时提高 GC 阈值，同时在 ~emacs-startup-hook~ 注册恢复函数；由 ~config.org~ 生成的 ~config.el~ 在正常配置结束时也主动恢复一次。恢复操作是幂等的，兼顾正常启动、手工加载和配置中途报错。

** D012：使用私有 GitHub 仓库管理配置

状态：已实现。

仓库为 ~auspicious-ssw/emacs-config~ ，可见性为私有。仓库只包含配置、文档和验证脚本；运行状态、缓存、已安装包、编译产物及密钥全部排除。

** D013：使用 Dashboard 作为启动首页

状态：已实现。

使用 MELPA 的 ~dashboard~ 包显示最近文件、书签和 Emacs 内置 ~project.el~ 项目，并使用成熟的 ~nerd-icons~ 包与 Symbols Nerd Font 提供标题和文件图标。普通 GUI 启动通过 ~dashboard-setup-startup-hook~ 打开首页，daemon/client frame 通过 ~initial-buffer-choice~ 打开首页。

不引入 Projectile、Doom Emacs、无正式 Release 的 Doom Dashboard 扩展或自定义首页内容渲染逻辑。首页内容与布局继续使用 Dashboard、Nerd Icons 的公开配置变量；键盘焦点只通过 D015 记录的 buffer-local hook 和选择 overlay 适配，不修改插件源码，保持升级边界清晰。

Dashboard 仍遵循离线启动约束：Dashboard 缺失时 Emacs 回到普通初始 buffer；Nerd Icons 包缺失时 Dashboard 退化为文字列表；字体缺失时图标可能显示为方框，并由本地检查与 GUI 验收发现。任何缺失都不会在启动时联网安装，用户稍后显式执行维护命令。

** D014：GUI 默认使用最大化普通窗口

状态：已实现。

初始 GUI frame 与 daemon/client 新建 frame 默认使用 ~fullscreen=maximized~ 。这里的“最大化”是占满当前显示器可用工作区的普通窗口，不是隐藏菜单栏和 Dock 的 macOS 原生全屏。

原因：用户日常将主工作窗口铺满可用工作区；相比写死像素坐标，最大化参数能自动适配显示器分辨率、菜单栏、Dock 和未来的多显示器变化。需要临时缩小时仍可使用 macOS 窗口按钮或 ~M-x toggle-frame-maximized~ 。

** D015：Dashboard 使用整项高亮表达唯一键盘焦点

状态：已实现。

Dashboard 隐藏只覆盖一个字符的原生光标，改用一个 Dashboard 专用选择 overlay 覆盖当前 widget 的完整文字范围。 ~r~、~p~、~m~、~TAB~ 和方向键移动 point 后，buffer-local ~post-command-hook~ 会同步移动这个 overlay，因此当前选中项始终以紫色背景完整高亮。

Dashboard widget 默认会在鼠标停留期间持续高亮条目；同时使用键盘时会形成两个视觉焦点。当前仍只清除 Dashboard 按钮 overlay 的 ~mouse-face~ ；手型指针、鼠标点击和按钮动作保持不变。选择 overlay 和更新 hook 都是 buffer-local，并在 Dashboard 每次重建时只保留一个实例，不修改插件源码，也不影响普通编辑 buffer。

** D016：使用 Doom Modeline 提供场景化信息底栏

状态：已实现。

使用独立的 ~doom-modeline~ 包替换 Emacs 默认 mode-line；这不安装或引入 Doom Emacs 配置发行版。选择原因是它直接复用现有 Nerd Icons，Batppuccin 已提供原生 face，并能根据 Dashboard、普通文件、Git、诊断和 LSP 等场景自动选择布局。

底栏遵循“信息必须支持当前判断或操作”的边界：

- 常驻显示文件/Buffer 状态、项目或路径、行列位置和主模式。
- Git 分支、诊断、LSP、远程主机、选区统计、宏录制和输入法仅在相关状态存在时显示。
- 隐藏 UTF-8 编码、换行符、次要模式列表、总行数、时间、电池、邮件和 GitHub 通知等低频或重复信息。
- 窗口变窄时由 ~doom-modeline-window-width-limit~ 自动收缩，不通过缩小字体塞入更多内容。

启动继续遵循离线约束：包缺失时保留 Emacs 默认 mode-line，不在启动期间联网安装。回滚时删除对应 ~use-package~ 配置和依赖声明，或直接 ~git revert~ 本决策对应提交。

** D017：使用 Magit 作为 Git 核心工作台

状态：已实现。

只安装 Magit 核心包，不同时引入 Forge、Projectile、diff-hl 或 Git 配置发行版。Magit 负责 Status、diff、stage/unstage、commit、log、branch、stash 与远端命令；Git 事实仍由仓库和 Git CLI 持有，Emacs 配置不维护第二份状态。

当前交互约定：

- ~C-x g~ 打开当前仓库状态页； ~C-c g~ 打开全局命令菜单； ~C-c f~ 打开当前文件菜单。
- ~C-x p m~ 从内置 ~project.el~ 进入当前项目的 Magit Status。
- Status 使用当前窗口，diff 与 process 可以使用另一窗口；按 ~q~ 恢复进入 Magit 前的窗口布局。
- 当前 hunk 按需显示词级差异，兼顾可读性和大型 diff 性能。
- 修改过的仓库 buffer 在 Git 操作前仍询问是否保存；自动 commit、push、rebase 和静默保存均未启用。
- ~magit-no-confirm~ 设为 ~nil~ ，保留危险操作确认，符合当前“先理解再操作”的阶段目标。

Magit 与 Transient 仍遵守离线启动约束：启动只注册 autoload 和键位，不刷新包索引，也不加载完整 Magit；菜单历史写入 =${XDG_STATE_HOME:-~/.local/state}/emacs/transient/=。包缺失时相关键位不注册，普通编辑、Dashboard 和内置 Git 命令不受影响。

Batppuccin 已提供 Magit 的 branch、diff、section、log、process 与 blame face，因此不额外覆盖插件 face。这样既保持当前视觉一致，也避免升级时维护一套重复颜色定义。

** D018：只把 Transient 命令菜单做成悬浮卡片

状态：已实现。

使用 ~transient-posframe~ 与其 ~posframe~ 依赖，把 ~C-c g~、~C-c f~ 以及 Magit 中 commit、push、branch 等 Transient 菜单显示为真正的 GUI child-frame。Magit Status、diff、log 和 process 继续使用 D017 的正常窗口布局，因为这些内容需要完整宽度、滚动和 section 交互，不适合塞进小悬浮窗。

视觉约定：

- 悬浮菜单以当前工作窗口为中心，继承 17pt Fira Code。
- 宽度下限为 84 字符，左右各保留 14px fringe，使用 2px 强调边框。
- 背景、文字和边框通过 Batppuccin 的公开 palette API 使用 Mocha mantle、text 与 mauve；不复制 Transient 自己已经由主题覆盖的 key、heading、argument face。
- 保持实色背景，不使用透明度，优先保证 Git 参数、快捷键与中文可读。

~transient-posframe-mode~ 是全局模式，而插件在 TTY 中不能创建 child-frame。配置不在启动时无条件启用它，而是在每次 ~transient-setup~ 前检查当前 frame：GUI 可用时启用，TTY、batch、daemon 的终端 client 或 child-frame 不可用时关闭并恢复 Transient 原生底部 side window。插件缺失时 Magit 本身继续正常工作。

启动仍然禁止联网：普通启动不加载 Transient、Posframe 或完整 Magit；它们只在首次使用 Magit/Transient 时按需加载。回滚时删除 ~transient-posframe~ 的 ~use-package~ 配置、同步函数与依赖声明，Magit 会直接恢复原生 Transient 菜单。

这里不再制作一个“SSW 悬浮菜单插件”。真正的 child-frame 创建、生命周期和 Transient 接管都由成熟的 ~transient-posframe~ / ~posframe~ 完成；本地函数只解决 GUI 与 TTY 共用配置时的逐 frame 开关策略。它不是可独立复用的功能边界，包装发布反而会增加版本和兼容维护。若适配代码明显增长，第一步是移入仓库内 ~lisp/core-git.el~ ，而不是立即发布新包。

** D019：使用 SIS + macism 只设置 GUI 启动输入源

状态：已实现。

使用独立的 SIS 插件连接 Emacs 与 macOS 输入源，底层调用本机已安装的 ~macism~ 。英文输入源固定为 ~com.apple.keylayout.ABC~ ，中文输入源记录为 ~com.apple.inputmethod.SCIM.ITABC~ ；GUI 应用从 Finder 启动时可能缺少 Homebrew PATH，因此配置保存已验证的 ~macism~ 绝对路径。

当前只在 ~emacs-startup-hook~ 和 ~after-make-frame-functions~ 调用一次公开命令 ~sis-set-english~ ：前者覆盖普通图形启动的首个 frame，后者覆盖 daemon/client 和后续新顶层 frame。Posframe、tooltip 等 child-frame 会被显式忽略；TTY、batch 与不可用环境也不修改系统输入源。

不启用 ~sis-global-respect-mode~、context、inline、cursor-color 或 auto-refresh。这样 Emacs 打开时默认是英文，但用户进入正文后手动切到中文会一直保留，不会因 prefix、buffer、光标位置或定时器被反复切走。SIS 或 macism 缺失时配置安全跳过这项增强，普通编辑仍可启动；回滚时删除 SIS 的 ~use-package~、两个 hook、适配函数和包声明即可。

** D020：仓库说明与设计文档统一使用 Org

状态：已实现。

根入口使用 ~README.org~ ，架构、决策、快捷键、运维和路线文档统一使用 ~docs/*.org~ 。所有内部导航使用 Org 文件链接，源码块、表格、行内代码和标题都使用原生 Org 语法，不保留内容重复的 Markdown 副本。

原因：Emacs 已成为主编辑器，Org 能直接提供折叠、结构编辑、源码块和后续学习系统衔接；单一格式也避免 Markdown 与 Org 两份说明在配置演进时互相漂移。GitHub 能直接渲染仓库中的 Org 文档，因此不需要额外生成一份 README。

** D021：使用 org-modern + org-appear 提升 Org 可读性

状态：已实现。

~org-modern~ 统一处理标题折叠标记、列表、复选框、表格、标签、时间戳和源码块； ~org-appear~ 在光标进入时恢复被隐藏的强调标记、链接目标和实体语法。正文继续使用 17pt Fira Code 与 PingFang SC 中文回退，标题只做相对字号和字重层级，保留 Batppuccin 配色。

选择这两个包是因为职责互补：前者提供结构视觉，后者解决“隐藏语法好看但难编辑”的问题。 ~org-modern~ 已完整覆盖 ~org-superstar~ 与 ~org-bullets~ 的职责，因此不叠加重复插件；也暂不启用 variable-pitch，避免表格、源码块和中英文混排的基线漂移。两个包只通过 ~org-mode-hook~ 启用，不定义 keymap；缺包时退回原生 Org，启动过程仍不联网。

验证包括：普通启动不提前加载 Org 美化包；打开 Org buffer 后两个 minor mode、行距和可读性选项生效；光标进入强调文本时隐藏标记恢复； ~TAB~、~C-c C-c~、~C-g~、~C-SPC~、~M-SPC~ 与 ~M-x~ 保持原语义。回滚时移除两个 ~use-package~ 块、~ssw/org-visual-setup~ 与包声明即可，Org 文档内容无需迁移。

** D022：使用 ~config.org~ 作为 Literate 配置唯一事实源

状态：已实现；Tangle 一致性、离线加载边界和完整回归以 ~scripts/check.sh~ 的当次输出为准。

人工维护的主配置从 ~init.el~ 迁移到 ~config.org~ ，按配置目的组织说明与 Emacs Lisp 源码块。 ~config.el~ 是确定性的 Tangle 产物并纳入 Git； ~init.el~ 只加载本地 ~config.el~ ， ~early-init.el~ 保持 D003 的早期职责。

选择这一结构的原因：

- 配置意图、约束和实现放在同一 Org 结构内，便于在 Emacs 中折叠、导航和学习。
- 跟踪 ~config.el~ 后，全新克隆、离线环境和缺少第三方 Org 外观包时仍能直接启动。
- 最小 ~init.el~ 不在每次启动时执行 Tangle，避免启动写文件、依赖源码时间戳或因 Org 解析错误扩大故障面。
- 人工入口只有 ~config.org~ ；生成物虽受 Git 管理，但只用于执行、差异审查和离线恢复，不形成第二份可独立演进的源码。

约束：

- 日常不得直接修改 ~config.el~ ；修改 ~config.org~ 后必须重新 Tangle，并同时审查两者差异。
- ~config.org~ 与 ~config.el~ 不一致时，使用 ~config.org~ 重新生成，不反向复制生成文件覆盖主配置。
- Tangle 必须使用 Emacs 自带 Org 即可完成，不要求先安装 ~org-modern~ 或 ~org-appear~ ，也不得联网。
- 提交前检查目标包括 Tangle 一致性、 ~init.el~ 只加载本地生成物，以及生成后的 batch/离线启动；实际通过状态只以当前验证输出为准。

回滚时应把 ~config.org~ 与 ~config.el~ 作为同一变更单元一起反向提交。若 Literate 入口本身阻断启动，可以从 Git 中已跟踪的上一版 ~config.el~ 恢复运行，再修复 ~config.org~ 并重新生成；不得把临时手改的生成物作为长期修复。
